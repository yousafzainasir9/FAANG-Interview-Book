Problem A: Longest Substring with At Most K Distinct Characters (20 min)

SLIDING WINDOW
Given a string s and integer k, return the length of the longest substring containing at most k distinct characters.

Example: s = "eceba", k = 2 → Output: 3 ("ece")

Solution Walkthrough
Use a sliding window with hash map to track character frequencies. Expand right pointer, contract left when distinct count exceeds k.
Sliding Window HashMap Two Pointers
A
1
Initialize window: Two pointers left=0, right=0. HashMap to track char frequencies in current window.
2
Expand window: Move right pointer, add char to map. Update max length if valid.
3
Contract when needed: If distinct chars > k, shrink from left until valid again.
4
Return max length: Tracking maximum valid window size throughout.
C# 
public int LengthOfLongestSubstringKDistinct(string s, int k) {
    if (k == 0 || s.Length == 0) return 0;

    var charCount = new Dictionary<char, int>();
    int left = 0, maxLen = 0;

    for (int right = 0; right < s.Length; right++) {
        // Add right char to window
        if (!charCount.ContainsKey(s[right])) {
            charCount[s[right]] = 0;
        }
        charCount[s[right]]++;

        // Shrink if too many distinct chars
        while (charCount.Count > k) {
            charCount[s[left]]--;
            if (charCount[s[left]] == 0) {
                charCount.Remove(s[left]);
            }
            left++;
        }

        // Update max length
        maxLen = Math.Max(maxLen, right - left + 1);
    }

    return maxLen;
}

O(n)
Space
O(k)
Approach

Problem B: Alien Dictionary (25 min)

GRAPH / TOPOLOGICAL SORT
Given a list of words sorted in an alien language, derive the order of characters in that language.

Example: words = ["wrt", "wrf", "er", "ett", "rftt"] → "wertf"

Solution Walkthrough
Build a directed graph from character ordering constraints. Use topological sort (DFS + visited set) to determine character order.
Topological Sort Graph DFS
B
1
Build graph: Compare adjacent words character-by-character. First difference = ordering constraint (edge).
2
Collect all chars: Add all characters from all words as nodes in graph.
3
Topological sort: DFS with visited tracking. Add to result after processing all neighbors (post-order).
4
Validate & return: Check for cycles. If cycle exists, invalid ordering (return "").
C# 
public string AlienOrder(string[] words) {
    var graph = new Dictionary<char, HashSet<char>>();
    var indegree = new Dictionary<char, int>();

    // Initialize all chars with indegree 0
    foreach (var word in words) {
        foreach (char c in word) {
            if (!indegree.ContainsKey(c)) {
                indegree[c] = 0;
                graph[c] = new HashSet<char>();
            }
        }
    }

    // Build graph: compare adjacent words
    for (int i = 0; i < words.Length - 1; i++) {
        string word1 = words[i], word2 = words[i + 1];
        int minLen = Math.Min(word1.Length, word2.Length);

        if (word1.Length > word2.Length && word1.Substring(0, minLen) == word2) {
            return ""; // Invalid: longer word before prefix
        }

        for (int j = 0; j < minLen; j++) {
            if (word1[j] != word2[j]) {
                if (!graph[word1[j]].Contains(word2[j])) {
                    graph[word1[j]].Add(word2[j]);
                    indegree[word2[j]]++;
                }
                break;
            }
        }
    }

    // Topological sort using Kahn's algorithm
    var queue = new Queue<char>();
    foreach (var kvp in indegree) {
        if (kvp.Value == 0) queue.Enqueue(kvp.Key);
    }

    string result = "";
    while (queue.Count > 0) {
        char c = queue.Dequeue();
        result += c;

        foreach (char neighbor in graph[c]) {
            indegree[neighbor]--;
            if (indegree[neighbor] == 0) queue.Enqueue(neighbor);
        }
    }

    return result.Length == indegree.Count ? result : "";
}

Coding Problem: Design LRU Cache (25 min)

LINKED LIST + DICTIONARY
Implement an LRU (Least Recently Used) Cache with O(1) get() and put() operations. Capacity is fixed. Evict least recently used item when full.

Methods: get(key) → value, put(key, value) → void

C# 
public class LRUCache {
    private int capacity;
    private Dictionary<int, Node> cache;
    private Node head, tail;

    private class Node {
        public int key, value;
        public Node prev, next;
        public Node(int k, int v) { key = k; value = v; }
    }

    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.cache = new Dictionary<int, Node>();

        // Sentinel nodes for easy removal
        head = new Node(0, 0);
        tail = new Node(0, 0);
        head.next = tail;
        tail.prev = head;
    }

    public int Get(int key) {
        if (!cache.ContainsKey(key)) return -1;

        Node node = cache[key];
        RemoveNode(node);
        AddToHead(node);
        return node.value;
    }

    public void Put(int key, int value) {
        if (cache.ContainsKey(key)) {
            Node node = cache[key];
            node.value = value;
            RemoveNode(node);
            AddToHead(node);
        } else {
            if (cache.Count == capacity) {
                RemoveNode(tail.prev);
                cache.Remove(tail.prev.key);
            }
            Node newNode = new Node(key, value);
            cache[key] = newNode;
            AddToHead(newNode);
        }
    }

    private void RemoveNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void AddToHead(Node node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }
}

O(1)
Space
O(capacity)

Follow-up Questions (15 min)

Q1: How would you handle concurrent access?

Add lock/synchronization for thread-safe operations. Consider ReaderWriterLockSlim for better read performance. Trade-off: complexity vs. concurrency.

Problem A: Binary Tree Right Side View (22 min)

BFS / LEVEL ORDER
Given a binary tree, return the values of nodes visible from the right side (rightmost node of each level).

Example: Tree [1,2,3,null,5,null,4] → [1,3,4]

C# 
public IList<int> RightSideView(TreeNode root) {
    var result = new List<int>();
    if (root == null) return result;

    var queue = new Queue<TreeNode>();
    queue.Enqueue(root);

    while (queue.Count > 0) {
        int levelSize = queue.Count;

        // Process all nodes at current level
        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.Dequeue();

            // Add rightmost node of level
            if (i == levelSize - 1) {
                result.Add(node.val);
            }

            // Add children for next level
            if (node.left != null) queue.Enqueue(node.left);
            if (node.right != null) queue.Enqueue(node.right);
        }
    }

    return result;
}

Problem B: Random Pick with Weight (23 min)

PREFIX SUM + BINARY SEARCH
Given an array of weights, implement pickIndex() that returns a random index weighted by the values.

Example: w = [1,3] → pickIndex() returns 1 with 75% probability, 0 with 25%

C# 
public class Solution {
    private int[] prefixSum;
    private Random random;
    private int totalWeight;

    public Solution(int[] w) {
        random = new Random();
        prefixSum = new int[w.Length];
        totalWeight = 0;

        // Build prefix sum array
        for (int i = 0; i < w.Length; i++) {
            totalWeight += w[i];
            prefixSum[i] = totalWeight;
        }
    }

    public int PickIndex() {
        // Generate random number in [1, totalWeight]
        int target = random.Next(1, totalWeight + 1);

        // Binary search for first index where prefixSum[i] >= target
        int left = 0, right = prefixSum.Length - 1;
        while (left < right) {
            int mid = left + (right - left) / 2;
            if (prefixSum[mid] < target) {
                left = mid + 1;
            } else {
                right = mid;
            }
        }

        return left;
    }
}

Problem: Meeting Rooms II (30 min)

MIN-HEAP / PRIORITY QUEUE
Given meeting times (start, end), find minimum number of conference rooms needed to host all meetings without overlap.

Example: [[0,30],[5,10],[15,20]] → 2 rooms needed

C# 
public int MinMeetingRooms(int[][] intervals) {
    // Sort meetings by start time
    Array.Sort(intervals, (a, b) => a[0].CompareTo(b[0]));

    // Min heap to track room end times
    var minHeap = new PriorityQueue<int, int>();
    minHeap.Enqueue(intervals[0][1], intervals[0][1]);

    // Process each meeting
    for (int i = 1; i < intervals.Length; i++) {
        // If earliest ending room is free, reuse it
        if (minHeap.Peek() <= intervals[i][0]) {
            minHeap.Dequeue();
        }

        // Add new meeting end time to heap
        minHeap.Enqueue(intervals[i][1], intervals[i][1]);
    }

    // Heap size = minimum rooms needed
    return minHeap.Count;
}

System Design Follow-up: Calendar System (15 min)

Q: How would you design a calendar system that schedules millions of meetings?

High-Level Design:
1. User Service: Authentication, user management
2. Calendar Service: Create/update/delete events
3. Meeting Room Service: Allocate rooms, check availability
4. Notification Service: Reminders via email/SMS
5. Database: PostgreSQL (events, users), Redis (room availability cache)

Q: What about conflict detection at scale?

Use optimistic locking with version numbers. When creating event, check version hasn't changed. If conflict, retry. For high-concurrency: use distributed locks (Redis) or event sourcing (Kafka log all updates, compute state).

Durability: Data survives hardware failures
URL expiration: TTL support (7 days, 1 year, never)

Step 2: High-Level Architecture (10 min)

System Architecture
Client → Load Balancer → API Gateway → Service Cluster (shorten, redirect) → Cache (Redis) → Database (PostgreSQL) → Blob Storage (optional)
★
1
Load Balancer: Nginx/HAProxy distributes traffic across API servers. Health checks every 5s.
2
API Service: Stateless microservices. Shorten (write) and Redirect (read) services separate.
3
Cache Layer: Redis (TinyURL → long URL). TTL matches database. 80% hit rate expected.
4
Database: PostgreSQL with read replicas. Sharding by hash(code) for horizontal scale.

Step 3: Deep Dive — Hashing Algorithm (20 min)

How to generate short codes?

Approach A: Counter + Base62
Generate unique auto-increment ID from database
Convert ID to base62 (0-9, a-z, A-Z)
Fast, deterministic, collision-free
Issue: Predictable (sequential codes)
Approach B: Random Base62
Generate random 6-8 character string
Check if exists in DB (collision handling)
Harder to reverse-engineer URLs
Issue: Collision overhead at scale

Recommended: Hybrid Approach
1. Use Zookeeper/Consul to allocate ID ranges to each server
2. Each server generates IDs locally (no DB bottleneck)
3. Convert to base62
4. Result: Predictable but distributed, no collisions, minimal latency

C# 
public string IdToBase62(long id) {
    const string chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    string result = "";

    while (id > 0) {
        result = chars[(int)(id % 62)] + result;
        id /= 62;
    }

    return result.Length == 0 ? "0" : result;
}

public long Base62ToId(string code) {
    const string chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    long id = 0;

    foreach (char c in code) {
        id = id * 62 + chars.IndexOf(c);
    }

    return id;
}

Step 4: Database Design & Scaling (10 min)

DATABASE SCHEMA
Shorten URLs Table

id (BIGINT, PK) | code (VARCHAR(8), UNIQUE, indexed) | long_url (TEXT) | created_at (TIMESTAMP) | expires_at (TIMESTAMP) | user_id (BIGINT, indexed) | click_count (INT)

Sharding Strategy:
Shard by hash(code) % 256. Each shard = separate database instance.
Write Path: API gets ID range from Zookeeper → converts to base62 → writes to shard → caches in Redis
Read Path: GET /{code} → check Redis → if miss, query appropriate shard → return 301 redirect

Bottleneck Analysis
Read hot URLs: Solution → Hot shard in separate cache, bloom filter
Database write bottleneck: Solution → Batch writes, use write-ahead log
Cache stampede: Solution → Cache warming, probabilistic early expiry
Zookeeper SPOF: Solution → Zookeeper cluster (3-5 nodes), consensus

Capacity Estimation

Metric Calculation Result
Daily New URLs 100M 1,157 req/s (peak 10K req/s)
Daily Redirects 1B 11,574 req/s (peak 100K req/s)
Storage (1 yr) 100M URLs × 500 bytes 50 GB (easily fits on 1 server)
Database Servers 100K writes/s → 10 servers per shard 256 shards × 10 = 2,560 servers
Cache (Redis) 80% hit rate, 24h expiry, ~50GB peak 2-3 Redis clusters (primary + replica)
⚠
Common Mistakes in System Design: Not asking clarifying questions, over-engineering (not needed at 100M scale), ignoring caching, not thinking about sharding key, forgetting monitoring/logging, assuming no data loss is acceptable.
