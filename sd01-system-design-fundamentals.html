How to Approach System Design Interviews

System design interviews test your ability to think about trade-offs, ask clarifying questions, and design solutions that scale. The key is a structured, step-by-step approach that demonstrates communication and technical depth.

The 4-Step Interview Framework

Every system design interview follows this pattern. Master it and you'll handle any problem:

Step 1: Requirements & Clarification (5 min)
Never assume. Ask questions to understand what you're building. Clarify functional requirements, non-functional requirements (scale, latency, availability), and constraints.
Functional Non-Functional Constraints
1

Key questions to ask:

  • Who uses this system? How many users? Growth trajectory?
  • What operations are most frequent (read-heavy, write-heavy, mixed)?
  • What are acceptable latency, throughput, and availability targets?
  • How much data do we expect to store?
  • Do we need real-time consistency or eventual consistency is fine?
  • Any regulatory, cost, or infrastructure constraints?
Step 2: High-Level Design (10 min)
Sketch the architecture at a 10,000-foot view. Draw boxes (services, databases, caches, queues) and arrows (data flow). Don't deep-dive yet—show you understand the big picture.
Architecture Components Data Flow
2

What to include:

  • Client layer: Web, mobile, API clients
  • API gateway: Entry point, rate limiting, authentication
  • Compute layer: Microservices, application servers (stateless)
  • Data layer: Primary database, replicas, cache, message queue
  • External services: CDN, third-party APIs, storage

Step 3: Deep Dive (25 min)
Pick the most important components and explain them in detail. Discuss data models, API contracts, scalability bottlenecks, and how you'd handle failures. Show you can zoom in without losing sight of the whole.
Data Models APIs Bottlenecks
3

Focus on:

  • Data schema: Tables, indexes, partitioning strategy
  • API endpoints: Method, path, request/response format
  • Traffic flow: Request journey through each component
  • Failure scenarios: What if a service fails? How do we handle it?
  • Scalability limits: Where does this break? Disk? CPU? I/O?
Step 4: Follow-up & Trade-offs (5 min)
Discuss alternatives you rejected and why. Address edge cases. Show you understand the problem space and can adapt. This is where senior thinking shines.
Tradeoffs Alternatives Robustness
4

Be ready to discuss:

  • Why did you choose SQL over NoSQL? (or vice versa)
  • Why Kafka instead of RabbitMQ?
  • How would you handle the "viral post" scenario?
  • What would you change if latency requirements were 10x stricter?
  • What if storage costs needed to be 50% lower?

What Interviewers Evaluate

Communication
Clarity & Reasoning
Explain your choices out loud. Don't assume knowledge. Walk the interviewer through your logic. Ask for feedback: "Does this direction sound good?"
Technical Depth
Know Your Tools
Understand databases, caches, queues, and APIs. Know their trade-offs. You don't need to implement, but you must know how they work.
Problem Solving
Bottleneck Thinking
Identify constraints. Where will your system break first? How would you scale past those limits? Show iterative improvement thinking.
ℹ Interview Length Breakdown (45 min total): Don't spend 20 minutes on high-level design. Move through it. Allocate time heavily to Step 3 where you show depth and handling trade-offs.

Scaling Fundamentals

Before choosing specific technologies, understand the fundamental patterns for scaling systems. These patterns apply across databases, services, and caches.

Vertical vs Horizontal Scaling

Horizontal Scaling (Scale Out)
Add more machines to the pool
Better for stateless services
Near-linear cost increase
Automatic with load balancers
Handles failures gracefully
Higher operational complexity
Vertical Scaling (Scale Up)
Add more CPU, RAM, disk to one machine
Limited by hardware ceilings
Downtime during upgrades
Single point of failure
Simpler operations initially
Exponential cost increase

Rule of thumb: Always design for horizontal scaling. Vertical is a short-term fix.

Stateless vs Stateful Services

Can your service lose its state without breaking the application?

Stateless
No local memory tied to a user or request. Each request contains all needed context (JWT, session ID). Easy to scale horizontally because any instance handles any request.
Examples: API servers, worker processes, stateless microservices
Stateful
Maintains in-memory state between requests. Requests must route to the same instance (sticky sessions). Harder to scale; requires session replication or external state store.
Examples: WebSocket connections, game servers, old servlet containers
⚠ Interview Tip: Design stateless where possible. If you must be stateful, externalize state to Redis or a database. This shows you understand distribution.

Monolith vs Microservices

The eternal trade-off. Both are valid; know when each makes sense.

Aspect Monolith Microservices
Deployment Single artifact; all or nothing Independent; can deploy service A without B
Scaling Scale entire app (wasteful if only one feature needs scaling) Scale only hot services
Latency In-process calls (very fast) Network calls (slower, needs retries)
Data Consistency Transactions easy Distributed transactions hard (eventual consistency)
Debugging Single codebase; request traces are simple Multi-service; need distributed tracing
Operational Overhead Lower (fewer moving parts) Higher (service discovery, orchestration, monitoring)
Team Size Works for small teams Shines with multiple teams (clear ownership)
✓ For interviews: Start with a monolith for system design questions unless scale demands otherwise. Microservices add complexity; don't default to them. If you do use them, explain why (independent scaling, team boundaries, technology diversity).

API Design Patterns

How do your services communicate?

Protocol Use Case Pros Cons
REST General-purpose web APIs Simple, cacheable, stateless, standard HTTP Verbose, over/under-fetching, poor real-time
GraphQL Flexible client queries Fetch exactly what you need, single endpoint Complex server, harder caching, learning curve
gRPC Service-to-service, high throughput Fast (binary), strongly typed, streaming, multiplexing Not human-readable, poor browser support, debugging harder
WebSocket Real-time bidirectional Low latency, persistent, full-duplex Stateful, harder load balancing, connection management

Interview strategy: REST is the default. Use GraphQL if the problem emphasizes flexible queries. Use gRPC if it's service-to-service and throughput matters. WebSocket for real-time features (notifications, chat, live updates).

Load Balancing

A load balancer distributes traffic across multiple servers, preventing any single server from becoming a bottleneck. It's a critical component in any scalable system.

Layer 4 vs Layer 7 Load Balancing

L4 (Transport Layer)
Routes based on IP protocol data (source/dest IP, port). Makes decisions without inspecting payload. Ultra-fast, handles any protocol.
Speed: Very fast | Example: Nginx (stream), HAProxy (mode tcp)
L7 (Application Layer)
Routes based on HTTP headers, paths, hostnames, cookies. Understands HTTP semantics. Slower but smarter routing possible.
Speed: Slower | Example: Nginx (http), HAProxy (mode http), AWS ALB

In practice: For most web services, use L7 (HTTP) load balancers. They let you route by path, hostname, or header—useful for A/B testing, canary deployments, and feature toggles.

Load Balancing Algorithms

Algorithm How It Works Best For Drawback
Round Robin Sends requests to servers in order: 1 → 2 → 3 → 1 Homogeneous servers, uniform load Ignores server capacity; no sticky sessions
Weighted Round Robin Favors capable servers (Server A gets 2x traffic) Heterogeneous servers (different specs) Still ignores current load
Least Connections Routes to server with fewest active connections Long-lived connections (e.g., WebSocket) Assumes connections have equal cost
Least Response Time Routes to server with lowest latency Variable response times Adds monitoring overhead
IP Hash Hash client IP → always same server Stateful services (sticky sessions) If a server dies, hashing breaks; uneven distribution
Consistent Hashing Hash request key → virtual ring Distributed caches, stateful services, minimal rebalancing Complex to implement and debug

Health Checks & Failover

A load balancer must detect failed servers and remove them from rotation.

Active Health Check
Load balancer periodically pings each server (e.g., HTTP GET /health). If no response, mark it down.
Pros: Quick detection | Cons: Overhead, tuning intervals is tricky
Passive Health Check
Monitor actual requests. If a certain % fail or timeout, mark server down.
Pros: No extra traffic | Cons: Slower to detect, clients hit dead servers

Best practice: Use both. Active checks catch issues fast; passive catches slow failures. Combine with circuit breakers in your client code.

ℹ Common Pitfall: Round-robin with stateless services is fine. But if you add caching in-process, suddenly you need consistent hashing. Always think about statefulness.

Caching Strategies

Caching is one of the highest-impact optimizations. It reduces latency and database load. But it's also a source of complexity and bugs. Know the patterns.

Cache-Aside (Lazy Loading)

The most common pattern. Application code checks cache first; if miss, query the database and populate cache.

Pseudocode
get_user(user_id):
  cached_user = cache.get(user_id)
  if cached_user != null:
    return cached_user

  user = database.get(user_id)
  cache.set(user_id, user, 3600)  // TTL: 1 hour
  return user

Pros: Simple, doesn't require cache warming, works for any data. Cons: Cache miss = slow read (need to hit DB), stale data possible.

Write-Through

On write, update cache AND database simultaneously. Reads always hit cache.

Pseudocode
update_user(user_id, new_data):
  cache.set(user_id, new_data)
  database.update(user_id, new_data)
  return success

Pros: Cache always fresh, data consistency. Cons: Write latency increases (wait for both cache and DB), extra writes if cache miss.

Write-Behind (Write-Back)

Write to cache immediately, asynchronously flush to database later. Fast writes, risk of data loss.

Pseudocode
update_user(user_id, new_data):
  cache.set(user_id, new_data)
  queue.enqueue({"op": "update_db", "data": new_data})
  return success  // Return before DB write!

// Background worker:
worker():
  while true:
    item = queue.dequeue()
    database.update(item)

Pros: Ultra-fast writes. Cons: If cache dies before flush, data is lost. Eventual consistency (risky for critical data).

Read-Through

Cache acts as a proxy. Application asks cache, cache handles DB lookups transparently.

Pros: Cleaner application code. Cons: Requires infrastructure support (e.g., DynamoDB DAX, AWS ElastiCache).

Redis vs Memcached

Feature Redis Memcached
Data Types Strings, lists, sets, hashes, sorted sets, streams String only
Persistence RDB snapshots, AOF log (durability) None (in-memory only)
Replication Master-slave, sentinel, cluster None (manual replication via consistent hashing)
Atomic Operations Yes (INCR, LPUSH, etc.) Limited (CAS for compare-and-swap)
Lua Scripting Yes (execute multi-step operations atomically) No
Use Cases Sessions, leaderboards, queues, pub/sub, rate limiting Simple cache (strings), session storage
Ease of Use Rich API, requires learning Simple, boring, reliable

Interview guidance: Use Redis for anything complex (leaderboards, rate limiting, queues). Use Memcached for dumb caching. For most modern systems, Redis is the default.

Cache Eviction Policies

When the cache is full, which data gets removed?

Policy Description Use Case
LRU (Least Recently Used) Evict item not accessed for longest time General caching; locality of reference
LFU (Least Frequently Used) Evict item accessed least often overall Hotspot data; some items always popular
FIFO Evict oldest item (insertion order) Simple, fast; rarely used
TTL (Time-To-Live) Expire item after set duration Temporary data (sessions, tokens)
Random Remove random item Baseline; better than nothing

Redis supports all these. Memcached uses LRU. In interviews, mentioning "we'll use LRU with a TTL" sounds good.

Cache Invalidation Strategies

Phil Karlton: "There are only two hard things in Computer Science: cache invalidation and naming things."

Expiration-Based
Set a TTL. Cache expires automatically. Simple, no extra logic. Risk: stale data if TTL too long.
Best for: Non-critical data, user sessions
Event-Based
On data write, explicitly invalidate related cache keys. Ensures freshness but requires coordination.
Best for: Critical data, master data

In practice: Use short TTLs (minutes) for most data. For critical paths, use event-based invalidation (e.g., publish to Kafka when user updates, cache listener invalidates). Combine both for safety.

CDN Caching

Cache static assets (images, CSS, JS) at the edge (geographically closer to users).

Key headers to control caching:

  • Cache-Control: public, max-age=31536000 — Cache for 1 year
  • Cache-Control: no-cache, must-revalidate — Validate every time
  • ETag — Version identifier; CDN checks if changed

CDNs (Cloudflare, Akamai, AWS CloudFront) reduce server load and latency for global users. Worth mentioning in any design with global scale.

✓ Interview winning move: "We'll use Redis for application caching with LRU eviction and a 1-hour TTL. For static assets, we'll front with a CDN. Hot keys (like trending posts) we'll cache with event-based invalidation."

Databases & Storage

Choosing the right database is critical. There's no one-size-fits-all. Know the trade-offs between relational, document, key-value, and graph stores.

SQL vs NoSQL

The foundational split.

SQL (Relational)
ACID transactions (strong consistency)
Fixed schema; joins across tables
Mature, well-understood
Scales down great; scales up hard
PostgreSQL, MySQL, Oracle
NoSQL (Non-Relational)
Eventual consistency; BASE semantics
Flexible schema; easier horizontal scaling
Better for unstructured data
No joins (denormalization required)
MongoDB, Cassandra, DynamoDB, Redis

NoSQL Types

Type Structure Examples Best For
Key-Value Simple map: key → value Redis, Memcached, DynamoDB Caching, sessions, counters
Document JSON-like documents; flexible schema MongoDB, CouchDB, Firestore User profiles, content, flexible queries
Column-Family Optimized for wide rows, time-series Cassandra, HBase, BigTable Analytics, metrics, time-series data
Graph Nodes and edges; relationship focus Neo4j, ArangoDB Social networks, recommendations, knowledge graphs
Search Full-text indexing Elasticsearch, Solr Search, filtering, text analytics

ACID vs BASE

ACID (SQL Default)
Atomicity: Transaction succeeds or rolls back completely. Consistency: Data always valid. Isolation: Concurrent transactions don't interfere. Durability: Committed data survives failures.
Strong consistency; slower writes
BASE (NoSQL Default)
Basically Available: Always responsive. Soft state: Data may be temporarily inconsistent. Eventually consistent: Will reach consistency given time.
Fast writes; eventual consistency

Sharding Strategies

When data doesn't fit on one machine, distribute it across multiple shards.

Strategy How It Works Pros Cons
Range Sharding Shard by value range (e.g., user IDs 0-999 → shard1, 1000-1999 → shard2) Easy to implement; range queries efficient Uneven distribution (hotspots); rebalancing hard
Hash Sharding Hash the key; result mod number of shards Distributes evenly; simple Range queries inefficient; rebalancing redistributes all data
Directory Sharding Lookup table maps key → shard Flexible; can rebalance without rehashing all data Lookup overhead; directory must be highly available
Consistent Hashing Hash key and shard IDs onto a virtual ring Minimal rebalancing when adding/removing shards Complex; uneven distribution without virtual nodes
⚠ Sharding Pitfall: Once you shard, cross-shard queries are expensive. Transactions spanning multiple shards are hard. Design your schema to avoid hot shards and cross-shard queries upfront.

Replication: Master-Slave vs Master-Master

Master-Slave
One master (writes), multiple slaves (read replicas). Writes go to master, replicate to slaves.
Cons: Failover requires promotion; single point of write failure
Master-Master
Multiple masters; writes to any master, replicate to all. More resilient but harder to keep in sync.
Cons: Conflict resolution required; eventual consistency

For interviews: Master-slave is simpler to reason about. Mention "we'll have read replicas for scaling reads" and the interviewer will nod. If asked about failover, mention Sentinel or automatic promotion.

Indexing: B-Tree vs LSM Tree

B-Tree (SQL)
Balanced tree structure; leaf nodes store data. Great for range queries and point lookups. Random I/O.
Databases: PostgreSQL, MySQL, Oracle
LSM Tree (Log-Structured Merge)
Writes to in-memory memtable, then flushed to disk in sorted order. Optimized for write-heavy workloads.
Databases: LevelDB, RocksDB, Cassandra, HBase

Database Selection Quick Reference

Requirement Database Choice Rationale
Structured data, ACID PostgreSQL, MySQL Relational model fits; strong consistency needed
Unstructured / JSON MongoDB, Firestore Flexible schema; easier development
Time-series metrics Cassandra, InfluxDB, Prometheus Column-oriented; optimized for time-range queries
Social graph, recommendations Neo4j Relationship traversal is fast in graphs
Full-text search Elasticsearch Inverted index; ranking; aggregations
Distributed, horizontal scale DynamoDB, Cassandra, MongoDB Built for sharding and replication
Real-time counters / sessions Redis In-memory; atomic operations

Message Queues & Event Streaming

Decouple services with asynchronous messaging. Queues enable fan-out, retries, and resilience. Essential for scaling systems.

Pub/Sub vs Point-to-Point

Point-to-Point (Queue)
Message from producer to one consumer. Consumer acknowledges; message is removed. Example: order processing.
Tools: RabbitMQ, SQS | Usage: Tasks, jobs
Pub/Sub (Topic)
Message from publisher to multiple subscribers. Each subscriber gets a copy. Example: broadcast notifications.
Tools: Kafka, Redis Pub/Sub | Usage: Events, notifications

Kafka vs RabbitMQ vs SQS

Feature Kafka RabbitMQ SQS
Type Event streaming (append-only log) Message broker (point-to-point + pub/sub) Managed queue (AWS)
Throughput Very high (millions/sec) High High (but API-bound)
Durability Log persisted; replay messages Messages can be persisted Durable (AWS manages)
Use Cases Analytics, event sourcing, data pipeline Task queues, microservices Decoupling, simple async
Complexity High (distributed, tuning) Medium Low (managed service)

Event Sourcing

Instead of storing current state, store all events (changes) that led to it. Replay events to reconstruct state.

Example: User Account
// Instead of storing current balance, store events:
"account_created" {"user_id": 1, "balance": 0}
"money_deposited" {"user_id": 1, "amount": 100}
"money_withdrawn" {"user_id": 1, "amount": 30}

// Current state: sum up events → balance = 70

Pros: Full audit trail; replay to any point; easy debugging. Cons: Complexity; eventual consistency; schema evolution hard.

CQRS (Command Query Responsibility Segregation)

Separate read and write models. Writes update an event log; reads query a materialized view (eventually consistent).

Example: Order service writes events; analytics service reads from a denormalized view updated asynchronously.

Benefits: Optimize reads and writes independently. Complexity: Eventual consistency; sync issues between models.

✓ Interview approach: Use Kafka/RabbitMQ to decouple services. Mention "we'll use Kafka for high-throughput event streaming" or "RabbitMQ for reliable task processing." Don't overcomplicate with event sourcing unless the problem specifically demands an audit trail.

CAP Theorem & Consistency Models

Every distributed system makes trade-offs. Understand CAP and consistency models to reason about these choices.

CAP Theorem Explained

A distributed system can guarantee at most two of three properties:

Consistency
All nodes see the same data at the same time. If you write X=5, every read returns 5.
Availability
System always responds to requests (no timeouts). Every request succeeds or returns a result.

Partition Tolerance: System works even if network is partitioned (nodes can't communicate). In modern distributed systems, partitions happen—you can't avoid them.

So the real trade-off is: CA vs CP vs AP?

Choice Sacrifice Examples When to Use
CP (Consistency + Partition) Availability Zookeeper, etcd, HBase Banking, critical data where consistency > uptime
AP (Availability + Partition) Consistency DynamoDB, Cassandra, MongoDB (eventual consistency mode) Social media, user-facing, fault-tolerant
CA (Consistency + Availability) Partition Tolerance Single-node systems, traditional SQL on LAN Rare in practice; assumes no network failures

Consistency Models

What does "consistent" actually mean? There are multiple flavors:

Model Description Example Scenario
Strong Consistency Write completes → all reads see it immediately Banking ATM: withdraw $100 → balance reflects instantly everywhere
Eventual Consistency Writes propagate eventually; reads may see stale data temporarily Twitter: tweet visible on some replicas in milliseconds, all within seconds
Causal Consistency Events with causality are ordered; unrelated events may be out of order Comment + reply: if I see a comment, I'll see replies in order
Read-Your-Write Client sees its own writes immediately; others may lag Email: after sending, I see the sent message; others might take seconds
Monotonic Reads Client's reads never go backwards in time Leaderboard: my view of my score never decreases as I refresh

Quorum Reads & Writes

In a replicated system, you can trade consistency vs latency using quorum-based reads/writes.

Concept
Let N = total replicas, W = write quorum, R = read quorum

For "strong consistency": W + R > N
  // If we write to W replicas and read from R, overlap is guaranteed
  // Example: 3 replicas, W=2, R=2 → 2+2 > 3 ✓

Example:
  Write to 2/3 replicas
  Read from 2/3 replicas
  → At least 1 replica got the write, so if we read from 2, we'll see it

Tuning: Tight quorum (W=1, R=3) = fast writes, slower reads. Loose quorum (W=3, R=1) = slower writes, fast reads. Choose based on your workload.

ℹ Interview Insight: "We'll use eventual consistency for the feed (AP system) because availability is critical. For financial transactions, we'll use strong consistency (CP system) because correctness matters more."

Rate Limiting & API Gateway

Control traffic to prevent abuse and ensure fair resource allocation. Rate limiting is essential for any user-facing API.

Rate Limiting Algorithms

Token Bucket
Bucket holds tokens. Each request consumes a token. Tokens are added at a fixed rate. If bucket is full, excess tokens are discarded. Supports bursts.
Most Popular Allows Bursts
1

How it works:

  • Bucket capacity: 100 tokens
  • Refill rate: 10 tokens/second
  • Burst: User can send 100 requests immediately (empties bucket)
  • Then limited to 10 requests/second

Best for: User-facing APIs, allowing occasional bursts.

Sliding Window Log
Track timestamp of every request. To check if request is allowed, count requests in the past window duration. If count < limit, allow it.
Accurate Memory Overhead
2

Example: Limit to 100 requests per minute.

  • Store timestamps of all requests in Redis
  • On new request: count entries in log from [now - 60s, now]
  • If count < 100: allow and add timestamp
  • Memory usage can be high (store per request)
Fixed Window
Divide time into fixed buckets (e.g., each minute is a bucket). Count requests in current bucket. Simple but has edge-case issues.
Simple Has Issues at Boundaries
3

Problem: Users can exploit window boundaries.

Example Issue
Limit: 100 requests/minute
Minute 1: 59 requests (allowed)
Boundary: 1:00 → 2:00
Minute 2: 100 requests at 1:59.9, 100 at 2:00.1 (total 200 in 0.2s!)
→ User exceeded limit but appears within rules

Best for: Simple use cases; less critical systems.

Implementing Rate Limiting

Where to rate limit?

  • API Gateway: Global limit (per user/IP/API key)
  • Service: Per-endpoint limit or resource-specific limit
  • Database: Query rate limit

Storage: Use Redis. Key = user ID, value = token count. Fast atomic operations.

Pseudocode: Token Bucket in Redis
check_rate_limit(user_id, limit, refill_rate):
  key = "rate_limit:" + user_id
  now = current_time()

  data = redis.get(key)
  if data == null:
    redis.set(key, {"tokens": limit, "last_refill": now}, "EX" 86400)
    return true

  // Refill tokens based on time elapsed
  elapsed = now - data.last_refill
  new_tokens = min(limit, data.tokens + elapsed * refill_rate)

  if new_tokens >= 1:
    redis.set(key, {"tokens": new_tokens - 1, "last_refill": now})
    return true

  return false  // Rate limited

API Gateway Functions

An API gateway sits between clients and services. It provides:

Function Purpose Example
Routing Route requests to correct service based on path/host /users/* → User Service, /posts/* → Post Service
Authentication Validate JWT, API keys, OAuth tokens Reject requests without valid token
Rate Limiting Enforce quotas per user/IP/key 1000 requests/hour per API key
Request/Response Transform Modify headers, body, compression Add correlation ID, decompress payload
Load Balancing Distribute to backend services Round-robin or least-connections
Circuit Breaking Stop routing to failing services If User Service is down, return error fast
Caching Cache responses (GET requests) Cache /public/posts for 5 minutes
Logging / Monitoring Log all requests, metrics Track latency, error rates per endpoint

Popular API gateways: Kong, AWS API Gateway, Nginx, Envoy, Istio (service mesh).

✓ Interview Point: "We'll put an API Gateway in front that handles authentication, rate limiting, and routing. This centralizes cross-cutting concerns and makes the backend services simpler."

Monitoring & Observability

You can't improve what you don't measure. Observability—the ability to understand system state from outside—is critical for operating distributed systems.

The Three Pillars: Metrics, Logs, Traces

Metrics
Numerical measurements over time. CPU usage, request latency, QPS. Aggregated (e.g., p99 latency). Cheap to store.
Tools: Prometheus, Grafana, Datadog
Logs
Timestamped text events. "User 123 logged in", "Database error: timeout". High cardinality; expensive to store.
Tools: ELK, Splunk, Loki, CloudWatch

Traces: Request journey across services. Shows latency at each hop. Essential for debugging distributed systems. (Tools: Jaeger, Zipkin, DataDog, X-Ray)

Key Metrics to Monitor

Metric What It Measures Target Value
Request Latency (p50, p95, p99) How long requests take p99 < 200ms (depends on requirement)
Queries Per Second (QPS) Throughput Depends on capacity planning
Error Rate % of requests failing < 0.1% for production
CPU Usage Processor utilization 50-70% (avoid 100% = no headroom)
Memory Usage RAM utilization 70-80% (watch for leaks)
Disk I/O Read/write throughput, latency Depends on workload
Database Connection Pool Active connections Alert if > 80% capacity
Cache Hit Rate % of cache hits > 70% (indicates working cache)
Availability Uptime % 99.9% (2.74 hours downtime/month)

SLAs, SLOs, and Error Budgets

SLA (Service Level Agreement)
Contract with users: "We guarantee 99.9% availability." If you miss it, there are consequences (credits, compensation).
Binding | External
SLO (Service Level Objective)
Internal goal: "We aim for 99.95% availability." Tighter than SLA; gives buffer for maintenance.
Internal | Aspirational

Error Budget: If SLA is 99.9%, you have 43 minutes of downtime per month. Budget it wisely: deployments, experiments, maintenance.

Alerting Strategy

Alert when something matters, not everything. Too many alerts = alert fatigue = nobody reads them.

  • Alert on symptoms, not causes. Alert on "error rate > 1%" not "disk usage > 50%"
  • Severity levels: Critical (page on-call), Warning (create ticket), Info (log only)
  • Set thresholds carefully. Too low = noise. Too high = you miss real issues.
  • Include runbooks. When alert fires, include a link to: "Here's how to debug/fix this."
⚠ Common Mistake: Alerting on every metric. Alert on business impact: "Can users complete sign-up?" not "Is Redis at 80% memory?" The latter is a symptom; the former is the actual problem.

Back-of-Envelope Estimation

System design requires estimation: How many servers? How much storage? How much bandwidth? Master the numbers and frameworks to ballpark these quickly.

Key Latencies to Memorize

Operation Latency Notes
L1 cache reference 0.5 ns Single CPU cycle
Branch mispredict 5 ns
L2 cache reference 7 ns 14x slower than L1
Main memory access 100 ns 200x slower than L1
SSD random read 16 μs 16,000 ns
HDD random read 2 ms 2,000,000 ns; avoid if possible
Network round trip (same DC) 0.5 ms Local LAN
Network round trip (across US) 50 ms Cross-country
Database read (indexed) 1 ms In-memory, or SSD with page cache
Database write 5 ms Disk I/O, fsync
Disk sequential read 2 MB/sec Modern HDD
Memory sequential read 4 GB/sec Modern CPU

Rule of thumb: Memory < SSD < Network < HDD. Avoid network and disk I/O.

Estimation Workflow

Example: Design Instagram

1
Clarify scale: 1M daily active users, 100M monthly. Assume 3:1 read:write ratio.
2
QPS calculation:
DAU = 1M users
Requests per user per day = 100 (scroll, like, comment, upload)
Total requests/day = 1M × 100 = 100M
Requests/second = 100M / 86,400 ≈ 1,157 QPS

Read QPS ≈ 1,157 × 0.75 ≈ 867
Write QPS ≈ 1,157 × 0.25 ≈ 289
3
Storage estimation:
Posts/day = 289 write QPS × 86,400 sec ≈ 25M posts/day
Average post metadata (id, user_id, timestamp, likes, comments) = 1 KB
Post images: avg 2 MB
Storage/post ≈ 2 MB + 1 KB ≈ 2 MB

Storage/day = 25M posts × 2 MB = 50 TB/day
Storage/year = 50 TB × 365 ≈ 18 PB (petabytes!)
4
Bandwidth estimation:
Outbound: 867 read QPS × avg 2 MB = 1.7 GB/sec
Inbound: 289 write QPS × avg 2 MB = 578 MB/sec

Need CDN to distribute images globally!
5
Server count:
1 server handles ~5K QPS
Total QPS needed = 1,157
Servers = 1,157 / 5,000 ≈ 1 (but account for 2x peak, replication)
→ Deploy 10-20 servers for redundancy

Database Size Estimation

Example: User Table
Users = 100M
Per user: id (8B) + email (50B) + name (50B) + profile (200B) = ~300B
Total = 100M × 300B = 30 GB

Indexes: email (8B + 300B pointer) × 100M ≈ 30 GB
Total ≈ 60 GB for user table (reasonable)

Scale:
- Fits on single SSD (2-4 TB)
- But use sharding: 100 shards → 600 MB each (fast)

Interview Estimation Checklist

ℹ Quick checklist:
  • QPS (peak)? DAU × requests/user / 86,400
  • Storage (1 year)? requests/day × avg size / (1-waste%)
  • Bandwidth? peak QPS × avg response size
  • Servers? peak QPS / throughput per server
  • Database shards? total data / shard size
  • Cache size? working set size (~20-30% of total)
  • Replication factor? 3x for durability

Common Estimation Mistakes

  • Forgetting peak: Average QPS is not peak. Peak is 2-10x higher. Design for peak.
  • Ignoring growth: "1M users today" will be 10M in 6 months. Plan ahead.
  • Overlooking metadata: A 1 MB post might also store 100 KB of metadata (comments, likes). Count it.
  • Not accounting for replication: 1 TB storage requires 3 TB with 3x replication.
  • Underestimating bandwidth: Bandwidth is often the bottleneck. CDN is non-negotiable at scale.
✓ Interview success: Walk through estimation step-by-step. Say your assumptions aloud: "I'm assuming 100 requests per user per day. Is that reasonable?" Interviewers appreciate the reasoning more than accuracy.
