URL Shortener (TinyURL)

Design a service that converts long URLs into short, shareable links with redirect capability, analytics, and high availability.

Requirements

ℹ️
Functional: Generate short URL, redirect to original, track clicks, support custom slugs, analytics dashboard. Non-functional: Highly available, sub-100ms latency, no single point of failure, support 10M+ URLs.

Capacity Estimation

QPS
100K RPS
Write:Read
1:100
Storage
5 PB (10y)
Cache Hit
90%+

High-Level Architecture

Client → Load Balancer → API Servers (Shorten/Redirect) → Cache Layer (Redis) → Database (Master/Slave) → Analytics Queue

Core Components

1
Hash Generation: Use base62 encoding of auto-increment ID or MD5 hash. Base62 (a-z, A-Z, 0-9) gives 62^6 ≈ 56 trillion combinations for 6-char URLs.
2
Collision Handling: With sequential IDs + zookeeper for ID generation, virtually zero collisions. For random hashes, use bloom filters or collision counter.
3
Read Optimization: Cache layer with TTL-based expiry. Redirect API checks cache first (Redis), then DB. 99.9% hit rate expected.
4
Analytics: Async event queue (Kafka) writes clicks to time-series DB (InfluxDB). Real-time dashboards query aggregated metrics.

Key Insights & Trade-offs

✓ Sequential IDs + Base62
Guaranteed no collisions
Simple, predictable growth
Easy to shard by ID range
✗ Random Hash Approach
Must handle collisions
Harder to predict growth
Less friendly to ranges

Scaling: Use Zookeeper for distributed ID generation. Shard URLs by ID range (1-1M to server1, 1M-2M to server2). Cache at CDN edge for extremely hot URLs. TTL-based expiry reduces cleanup complexity.

Requirements

ℹ️

Fan-Out Strategy Comparison

✓ Fan-Out on Write (Push)
Fast feed read (pre-computed)
Good for average users
Cache is warm on read
✗ Fan-Out on Read (Pull)
Slow for celeb (1M+ followers)

Hybrid Approach (Recommended)

1
Push for Regular Users: User posts → enqueue to Kafka → Fan-out service pushes to Redis feed list for each follower. TTL 7 days.
2
Pull for Celebrities: Users with >1M followers trigger pull mode. Feed merge fetches from multiple celebrity timelines on read.
3
Ranking Engine: ML model scores tweets by engagement, recency, personalization. Top-K selection at read time.
4
Caching: Hot feeds cached in Redis, LRU eviction, bloom filters for missing tweets detection.

Architecture

Client → Load Balancer → Upload Service → Object Storage (S3) → Thumbnail Queue (Kafka) → Thumbnail Service → CDN → Global Users

Core Components

Object Storage: S3-compatible (AWS S3, GCS, Ceph). Partitioned by user_id/date for fast retrieval. Replication for durability.

Thumbnail Generation Pipeline

Pseudocode 
// Kafka consumer processes upload events
function processPhotoUpload(event) {
  const photo = event.photo;
  const s3Key = photo.s3_key;

  // Fetch original from S3
  const originalImage = s3.getObject(s3Key);

  // Generate thumbnails in parallel
  const sizes = [360, 720, 1080];
  const thumbnails = Promise.all(
    sizes.map(size =>
      resizeImage(originalImage, size)
        .then(img => s3.putObject(
          s3Key.replace('.jpg', `_${size}w.jpg`), img
        ))
    )
  );

  // Update metadata with thumbnail URLs
  db.updatePhoto(photo.id, {
    thumbnail_urls: {
      small: `cdn.domain/${s3Key}_360w.jpg`,
      medium: `cdn.domain/${s3Key}_720w.jpg`,
      large: `cdn.domain/${s3Key}_1080w.jpg`
    }
  });
}

Key Insights

Image Optimization: Use JPEG with 80% quality + WebP fallback for modern browsers. Reduces size by 30-40%. CDN handles format negotiation via Accept header.

Tag Search: ElasticSearch indexes tags separately for full-text search. Redis caches top 1000 tags for typeahead.

WhatsApp/Chat System

Design a real-time messaging system supporting 1:1 and group chats with message delivery guarantees, offline storage, and typing indicators.

Requirements

ℹ️
Functional: Send/receive messages, delivery status (sent/delivered/read), group chats, typing indicators, offline message queue. Non-functional: <100ms latency, exactly-once delivery, support 1B users, message history.

Capacity Estimation

Messages/Day
2B
Peak QPS
50K RPS
Concurrent Connections
100M
Avg. Msg Size
500 B

Architecture Overview

Client (WebSocket) → Connection Manager → Message Queue (Kafka) → Storage (Cassandra) → Delivery Service → Recipient

Core Components

1
Connection Manager: WebSocket servers handle persistent connections. Each server manages 10K-100K connections. Horizontal scaling via connection pooling.
2
Message Delivery: Kafka topics by recipient (topic: user_123). If online, push directly to WebSocket. If offline, store in Cassandra for retrieval on reconnection.
3
Exactly-Once Delivery: Message deduplication via idempotency keys. Cassandra stores message IDs with 7-day TTL to detect replays.
4
Read Receipts: Separate topic for delivery status. Aggregated in Redis for fast queries. Bloom filters detect read status.

Database Schema

  delivery_status TEXT,     -- 'sent', 'delivered', 'read'

CREATE TABLE user_chats (

  localStorage.save(message);

  // Step 2: Send to server
  websocket.send({
    type: 'MESSAGE',
    payload: message
  });

  // Step 3: Server writes to Kafka (with dedup)
  // Step 4: If recipient online, push via WebSocket
  // Step 5: Sender receives 'sent' ack
  // Step 6: Recipient receives message, auto-sends 'delivered' ack
  // Step 7: Sender sees 'delivered' badge
}

onMessageReceived(message) {
  // Deduplicate using idempotency key
  if (dedupeCache.has(message.idempotency_key)) {
    return;  // Already processed
  }

  // Store message
  db.saveMessage(message);

  // Update unread count
  cache.increment('unread:' + recipientId, 1);

  // Send delivery ack
  sendDeliveryAck(message.id, 'DELIVERED');
}

Typing Indicators & Presence

Implement typing indicator via WebSocket events (ephemeral, not stored). Presence updates use Redis pub/sub with 30-second heartbeats. Threshold of 3 missed heartbeats marks user offline.

Offline Message Handling

When user reconnects, fetch undelivered messages from Cassandra (sorted by timestamp). Batch fetch to avoid overload. Limit to last 100 messages or 30 days, whichever is smaller.

Driver Matching Algorithm

1
Candidate Generation: Query drivers within 2km radius using geohash neighbors + Redis GEORADIUS. Filter by availability.
2
Scoring: distance (40%), ETA to pickup (30%), driver rating (20%), surge multiplier (10%).
3
Request to Top 3: Send ride offer to top-3 candidates in parallel. First acceptance wins (30s timeout).
4
Trip Creation: Create trip record, stream real-time location updates to rider and driver via WebSocket.

ETA Calculation

Use Google Maps Distance Matrix API or internal routing service. Cache ETA for common routes (Redis). Adjust for real-time traffic (integrate Waze API). Server-side ETA updates every 30 seconds during trip.

Pricing Model

Fare = base (1.50) + distance_km * rate (0.25) + duration_min * rate (0.10) + surge_multiplier. Surge calculated per geohash (demand/supply ratio). Cap surge at 4x during peak hours.

Netflix/Video Streaming

Design a video streaming platform with adaptive bitrate delivery, global CDN, transcoding pipeline, and personalized recommendations.

Requirements

ℹ️
Functional: Stream video, pause/resume, recommendations, watch history, offline download. Non-functional: <2s startup, adaptive quality (480p-4K), 99.99% uptime, global availability.

Transcoding Pipeline

1
Master Upload: Content team uploads 4K video (200 Mbps, ProRes/DNxHD codec).
2
Chunk & Queue: Split into 10-second segments. Enqueue to transcoding farm with priority (popular shows first).
3
Parallel Transcode: Generate 5 bitrates (480p@1Mbps, 720p@3Mbps, 1080p@5Mbps, 1440p@15Mbps, 4K@25Mbps) using FFmpeg.
4
Packaging: HLS (HTTP Live Streaming) format with .m3u8 manifests. DASH as fallback. Cache-Control: 1 year (immutable).
5
CDN Distribution: Push to S3 + CloudFront edge locations. Replicate to Akamai, Cloudflare. Geo-pin to closest region (latency <50ms).

Adaptive Bitrate Selection

Client Algorithm 
selectBitrate() {
  const bw = measureBandwidth();
  const buffer = player.bufferSeconds;
  const options = [1, 3, 5, 15, 25];  // Mbps

  // Don't exceed 1.5x measured bandwidth
  let max = bw / 1.5;

  // Step down if buffer critically low
  if (buffer < 10) max = 3;

  // Cap mobile at 1080p
  if (isMobile) max = 5;

  return options.filter(b => b <= max).pop();
}

Recommendation Engine

Collaborative Filtering: User-item matrix from watch history. Compute cosine similarity with similar users. Recommend titles watched by similar cohorts.

Content-Based: Feature embeddings (genre, director, actors). Recommend semantically similar titles.

Ranking: Blend 70% collaborative + 30% content-based. Add trending boost (last 7 days views).

Google Search Engine

Design a search engine with web crawler, inverted index, PageRank algorithm, and query processing with spelling correction.

Requirements

ℹ️
Functional: Crawl web, index pages, full-text search, ranking by relevance, spelling correction. Non-functional: <100ms query latency, index billions of pages, 99.9% uptime.

Web Crawler Architecture

Seed URLs → URL Frontier (priority queue) → Downloader Pool (polite, 1-3s delays) → Parser (extract text, links) → Inverted Indexer (update posting lists)

URL Frontier (Distributed Priority Queue)

Kafka-based queue prioritized by PageRank, last-crawl-time (freshness), domain importance. Per-domain sub-queue enforces politeness (max 1 req/3s per domain). Bloom filter prevents duplicate URLs.

Inverted Index Structure

Index Format 
// Inverted index: word → posting list
word: 'search'
posting_list: [
  {
    doc_id: 123456,
    positions: [0, 45, 102],
    term_frequency: 3,
    field: 'body'
  },
  {
    doc_id: 789012,
    positions: [12, 78],
    term_frequency: 2,
    field: 'title'
  }
]

// Sharding: word_hash % num_shards determines shard server
// Each shard replicated on 3 servers for high availability

PageRank Computation

Iterative algorithm: PR(A) = (1-d)/N + d*(PR(B)/OutboundLinks(B) + ...). Damping factor d=0.85. Run weekly using MapReduce on entire web graph (100B+ nodes). Converges in ~20 iterations. Store precomputed scores in index.

Query Processing Pipeline

1
Parse & Clean: Extract keywords, operators (AND/OR/NOT), exact phrases. Remove stop words (the, a, is).
2
Spelling Correction: Edit distance ≤2 against dictionary. Use context (previous queries) for smart suggestions.
3
Index Lookup: Fetch posting list from shards (parallel). Merge (intersection for AND, union for OR).
4
Ranking: Score = TF-IDF(term, doc) * PageRank(doc) * Freshness(doc). Return top-10 results.

Rate Limiter

Design a distributed rate limiting system supporting token bucket, sliding window, and per-user limits with sub-millisecond latency.

Requirements

ℹ️
Functional: Enforce rate limits (RPS), per-user/IP/API key, configurable limits, graceful degradation. Non-functional: <1ms latency, distributed, no false negatives.

Token Bucket Algorithm

Algorithm 
// Redis-based token bucket
allowRequest(userId, limit, refillRate) {
  const key = `rate:${userId}`;

  // Get current bucket state
  const bucket = redis.hgetall(key);
  const tokens = bucket.tokens || limit;
  const lastRefill = bucket.lastRefill || now();

  // Refill tokens based on time elapsed
  const elapsed = (now() - lastRefill) / 1000;
  const refilled = Math.min(limit, tokens + elapsed * refillRate);

  // Try to consume 1 token
  if (refilled >= 1) {
    redis.hset(key, {
      tokens: refilled - 1,
      lastRefill: now()
    });
    redis.expire(key, 3600);
    return true;
  }

  return false;
}

Sliding Window Log (Alternative)

Store request timestamps in Redis sorted set. Count requests in last 60 seconds (ZCOUNT). Simple but higher memory usage. Better for low RPS limits. Trade-off: precision vs memory.

Distributed Rate Limiting

Each API server has Redis connection. Token bucket state stored in Redis (single source of truth). Race condition resolved by Redis atomic operations (Lua script). Cluster: Redis Cluster with consistent hashing by user_id.

Lua Script for Atomic Operation

Lua 
-- Atomic token bucket refill + consume
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local refillRate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local bucket = redis.call('HGETALL', key)
local tokens = tonumber(bucket[2]) or limit
local lastRefill = tonumber(bucket[4]) or now

-- Refill
local elapsed = (now - lastRefill) / 1000
local refilled = math.min(limit, tokens + elapsed * refillRate)

-- Consume
if refilled >= 1 then
  redis.call('HSET', key, 'tokens', refilled - 1, 'lastRefill', now)
  redis.call('EXPIRE', key, 3600)
  return 1
end

return 0

Response Headers

X-RateLimit-Limit: 1000, X-RateLimit-Remaining: 245, X-RateLimit-Reset: 1234567890. Client can backoff proactively when approaching limit. Return 429 (Too Many Requests) when limit exceeded. Include Retry-After header.

Distributed Cache (Redis/Memcached)

Design a highly available distributed cache system with consistent hashing, replication, eviction policies, and cache warming.

Requirements

ℹ️
Functional: Get/Set/Delete, TTL expiry, eviction on memory full. Non-functional: <1ms latency, multi-node, no data loss, high throughput (1M+ ops/sec).

Consistent Hashing

Map keys to cache nodes using consistent hash function (MD5, SHA1). Hash ring with virtual nodes (150-200 per server) ensures balanced distribution. Node addition/removal only rehashes ~33% of keys (vs ~100% with modulo hashing).

Pseudocode 
// Consistent hashing with virtual nodes
initHashRing(servers) {
  const ring = {};  // hash -> server mapping

  for (const server of servers) {
    for (let i = 0; i < 150; i++) {
      const virtualKey = `${server.id}:${i}`;
      const hash = md5(virtualKey);
      ring[hash] = server;
    }
  }

  return ring;
}

getServer(key, ring) {
  const hash = md5(key);
  // Find next hash >= key hash (clockwise)
  const hashes = Object.keys(ring).sort();
  const idx = binarySearch(hashes, hash);
  return ring[hashes[idx % hashes.length]];
}

Eviction Policies

Policy When Pros Cons
LRU (Least Recently Used) Memory full Good temporal locality O(1) but requires tracking LRU list
LFU (Least Frequently Used) Memory full Evicts cold items Higher CPU overhead
FIFO (First In First Out) Memory full Simple, predictable Ignores access patterns
Random Memory full Minimal overhead Poor hit ratio

Replication & Failover

Each cache node replicated on 2-3 secondary nodes. Master writes go to all replicas (synchronous for critical data). Read from any replica. On master failure, elect new master from replicas (Raft consensus). Sentinel monitors health (ping, timeout 3s).

Cache Warming Strategy

On deployment, preload hot keys from database. Use offline analysis (access logs) to identify top 1000 keys. Batch load in background before marking cache ready. Prevents "thundering herd" on cold start (all clients query DB simultaneously).

Cache Invalidation Patterns

Pseudocode 
// Write-through: update cache + DB atomically
setWithWriteThrough(key, value) {
  cache.set(key, value, ttl=3600);
  db.update(key, value);
}

// Write-back: update cache, async DB write
setWithWriteBack(key, value) {
  cache.set(key, value, ttl=3600);
  queue.enqueue({type: 'WRITE', key, value});
  // Background worker processes queue
}

// Cache invalidation via versioning
deleteWithVersioning(key) {
  const versionKey = `${key}:version`;
  const version = cache.increment(versionKey);
  // Next read will see new version, old cached data ignored
}

Monitoring & Metrics

Track: hit rate (should be 90%+), miss rate, eviction count, memory usage, latency (p50, p99). Set alerts if hit rate drops below 85% (indicates insufficient cache size or ineffective policies). Use Redis INFO command for stats.
